# **DST OPERATOR PROTOCOL**

## **Clean Procedural Steps with Explanatory Notes**

# **0\. Core Principles (Direct Statements)**

**The kernel is the evolving instrument of DST.**

**The traversal report is the final product.**

The kernel is revised at each iteration based on jury findings and operator decisions.

Its purpose is to become precise enough to generate a correct traversal in a single one‑shot pass.

The traversal report produced by the final kernel — from a fresh project thread, using the declared canon and intake — is the actual deliverable of a DST project.

The audit trail documents how the kernel evolved and why the final traversal is reproducible and audit‑survivable.

# **Phase 1 — Project Setup**

### **Step 1\. Prepare all files locally.**

Download the kernel, canon documents, prior DST records, and all supporting materials into a local directory.

### **Step 2\. Create a new Claude project.**

Create a new Claude project and name it meaningfully.

### **Step 3\. Upload the kernel into the project’s Instructions panel.**

Place the DST kernel into the project’s Instructions section.

### **Step 4\. Upload all canon documents into the project’s Sources panel.**

Upload all canon documents into the project’s Sources section.

# **Phase 2 — Advisor Setup and Intake Construction**

### **Step 5\. Open a persistent Advisor thread in a separate Claude chat.**

Open a new Claude chat in a separate tab or window. This will serve as the Advisor.

### **Step 6\. Construct the intake with the Advisor.**

With the Advisor, formalize the problem into a complete intake, including the problem statement, representational spaces, relevant domains, formal base, nullification conditions, solution criteria, and any constraints required by the domain.

# **Phase 3 — Traversal Cycle**

### **Step 7\. Open a new, empty project chat window and fire the traversal.**

Inside the Claude project, open a brand‑new empty chat window. Paste the intake as the first message and send it.

### **Step 8\. Transfer the traversal report to the Advisor for analysis.**

Copy the traversal output and paste it into the Advisor thread.

### **Step 9\. Prepare the Jury Prompt with the Advisor.**

Using the Advisor, prepare a Jury Prompt. Use the generic jury protocol unless the traversal output contains specific issues requiring targeted attention.

### **Step 10\. Open a fresh Jury thread and submit the traversal for audit.**

Open a new, empty ChatGPT (or other LLM) thread. Paste only the Jury Prompt and the traversal report.

### **Step 11\. Transfer the Jury report to the Advisor for interpretation.**

Copy the jury’s report and paste it into the Advisor thread.

The jury identifies the blockages. The operator accepts these findings.

The Advisor helps the operator understand them; it does not classify or alter them.

### **Step 12\. Update the kernel with new generalized meta-rules.**

Using the jury’s identified blockages (and any self‑audit findings), the operator converts valid issues into generalized preventive meta‑rules and updates the kernel in the project’s Instructions panel.

### **Step 13\. Begin the next traversal cycle.**

**Update the project’s Instructions panel with the newly revised kernel.**

**Open a new, empty project chat window and fire a new traversal using the same intake as before.**

**This produces the next traversal report.**

# **Phase 4 — Stopping Conditions**

Stopping conditions include (examples, not constraints):

1. A valid proof is produced.  
2. A valid nullification or refutation is produced.  
3. A bridge direction is identified requiring a new intake.  
4. Disciplined Departure is triggered by the kernel’s criteria.  
5. The problem or domain specification requires re‑formalization.  
6. The traversal space is exhausted under the declared domain constraints.

DST is open‑ended; novel stopping modes may appear.

# **Summary: Why Fresh Threads and One‑Shot Passes Matter**

DST is designed to refine the kernel until it can produce a correct traversal report in a single one‑shot pass.

To achieve this, each traversal must be executed in a clean, empty project chat window with no conversational memory.

Each jury audit must be executed in a fresh, empty external thread with no prior context.

This ensures:

* no drift  
* no contamination  
* no hidden state  
* no accumulated bias  
* no reliance on conversational buildup

The final traversal report — produced by the final kernel, from a fresh instance — is the actual deliverable of a DST project.

The audit trail documents how the kernel evolved and why the final traversal is reproducible and audit‑survivable.
